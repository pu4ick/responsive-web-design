### Code Review

**1. Глобальные селекторы (Critical)**
Ты стилизуешь теги напрямую: `section {}`, `button {}`.
*   **Почему это плохо:** Этот CSS применится ко *всем* секциям и кнопкам на сайте. Ты сломаешь футер, другие статьи и сайдбары.
*   **Решение:** Всегда используй классы. В идеале — методологию **BEM** (Block Element Modifier). Например: `.hero`, `.hero__title`, `.btn`.

**2. Избыточный Flexbox на мобильных**
```css
/* Твой код для мобильных */
section {
  display: flex;
  flex-direction: column; /* Лишнее */
}
```
Блочные элементы (`div`, `h1`, `p`) и так ведут себя как "column" — они встают друг под другом. Тебе не нужен Flexbox, чтобы просто расположить блоки вертикально.
*   **Best Practice:** Оставь на мобильном стандартный поток (Normal Flow), а `display: flex` включай *только* внутри медиазапроса для десктопа. Чем меньше кода браузеру нужно парсить на телефоне, тем быстрее рендеринг.

### Refactoring (Как это пишут в Production)

Изучи этот код. Обрати внимание на классы и то, как мы "включаем" сетку только там, где она нужна.

**HTML:**
```html
<section class="hero">
  <div class="hero__content">
    <h1 class="hero__title">Управляйте продажами легко</h1>
    <p class="hero__desc">Лучшая CRM для малого бизнеса с аналитикой и интеграциями.</p>
    <!-- Кнопка действия (CTA) -->
    <button type="button" class="btn btn--primary">Попробовать бесплатно</button>
  </div>
  
  <!-- Картинка может быть декоративной, тогда alt пустой, 
       но если она несет смысл — описываем -->
  <img src="dashboard-preview.jpg" alt="Интерфейс CRM системы" class="hero__image">
</section>
```

**CSS (SCSS-like logic):**
```css
/* --- Base Styles (Mobile First) --- */
/* Никаких медиазапросов. Это стили для iPhone SE и старых Android */

.hero {
  padding: 2rem 1rem;
  text-align: center;
  /* Мы НЕ пишем display: flex здесь. 
     Пусть блоки встают друг под другом естественно. */
}

.hero__title {
  font-size: 1.5rem;
  margin-bottom: 1rem;
  line-height: 1.2;
}

.hero__desc {
  margin-bottom: 1.5rem;
  color: #555;
}

.hero__image {
  max-width: 100%; /* Чтобы картинка не порвала экран мобилки */
  height: auto;
  margin-top: 2rem; /* Отступ сверху на мобильном */
}

.btn {
  display: inline-block;
  width: 100%; /* На мобильном кнопку удобно нажимать пальцем во всю ширину */
  padding: 1rem;
  background-color: blue;
  color: white;
  border: none;
  cursor: pointer;
}

/* --- Desktop Styles (Progressive Enhancement) --- */
/* Усложняем макет только если экран позволяет */

@media (min-width: 768px) {
  .hero {
    display: flex; /* ВКЛЮЧАЕМ сетку только сейчас */
    align-items: center; /* Центрируем вертикально */
    justify-content: space-between; /* Разносим по краям */
    text-align: left; /* Возвращаем текст влево */
    padding: 4rem 2rem;
  }

  .hero__content {
    width: 50%; /* Занимаем половину экрана */
    padding-right: 2rem; /* Воздух между текстом и картинкой */
  }

  .hero__title {
    font-size: 3rem;
  }

  .hero__image {
    width: 50%;
    margin-top: 0; /* Убираем отступ, он больше не нужен */
  }

  .btn {
    width: auto; /* Возвращаем кнопке естественный размер */
  }
}
```

### Защита решения (Почему так?)

1.  **Производительность (Reflow/Repaint):** На мобильном телефоне мы не заставляем браузер просчитывать Flex-контейнер. Мы используем нативный поток документа (`display: block` по умолчанию). Это самый дешевый для процессора способ отрисовки.
2.  **Специфичность (Specificity) и Классы:** Использование `.hero__title` вместо `h1` позволяет нам безопасно менять стили заголовка только в этом блоке, не ломая заголовки в статьях блога.
3.  **Override (Переопределение):** Внутри `@media` мы переопределяем только то, что меняется. Например, `margin-top` у картинки нужен на мобильном (чтобы отделить от текста сверху), но не нужен на десктопе (так как она сбоку).
